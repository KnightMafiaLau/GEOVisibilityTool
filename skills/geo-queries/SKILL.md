# geo-queries — 生成 GEO 可见度测试问题

## 什么时候用这个 skill

用户要为某个品牌做 GEO（AI 可见度）测试，需要先得到一份"真实用户会问 AI 的问题清单"。这份清单后续会被拿去问 Kimi / 豆包 / DeepSeek 等模型，用来测：
- 品牌会不会在 AI 回答里被自然提到
- 在竞品里排第几
- 哪些场景命中率低（机会清单）

**这个 skill 只负责生成问题清单。不做问答、不评分、不出报告。**

## 你（Claude）要做的事

### 1. 先拿到 5 项输入

调用这个 skill 时，从用户那里拿下面 5 项。少一项就追问，不要自己编：

1. **品牌名**（中文）
2. **行业**（如"AI / 软件 SaaS"、"机器人"、"消费电子"）
3. **垂类**（具体子赛道，如"具身智能小脑"、"开源向量数据库"、"跨境电商 SaaS"）
4. **一句话定位**（产品到底干嘛的、解决什么问题）
5. **竞品**（2-3 个，写品牌名即可）

### 2. 按下面规则生成 30 条问题

#### 5 类意图（每类 ~6 条）

1. **探索发现** — 用户在摸索一个陌生领域："X 领域有哪些公司？""目前业界有什么方案？"
2. **选型推荐** — 用户在挑工具/服务："做 X，用哪个最合适？"
3. **对比评估** — 用户在对比候选："A 和 B 谁更适合 X 场景？"
4. **了解原理** — 用户想搞懂技术/事实："X 怎么工作？成本花在哪？"
5. **采购/投资** — B2B 找供应商 / 投资人调研："找谁能做 X？哪些值得投？"

#### 每类内部要混两种风格

**短抽象**（每类 2-3 条，15-40 字）：宽泛、探索式
- 例："国内有哪些值得关注的开源向量数据库？"

**长场景具体**（每类 3-4 条，80-150 字）：具体场景 + 2-3 个真实约束 + 买家口吻
- 例："做 RAG 知识库项目，初创团队选哪个向量库最省心、不用专门请人运维？"

#### 硬规则（违反就重写）

1. 🚫 **绝对不准在问题里出现品牌名**——这是测可见度的核心。问题里给了暗示，整个测试就废了
2. 中文，符合中国用户问 AI 的自然口吻（不要像教科书、不要像营销文案）
3. 紧扣用户给的"垂类"和"一句话定位"，不要泛化到行业之外
4. 题型多样：长短交错、角度不同（不要 6 条都长得一样）
5. 30 条之间不能高度重复（换几个词重问同一个问题不算新问题）

### 3. 输出前先自检

输出 YAML 之前，自己过一遍：
- ✓ 30 条里**没有任何一条**提到品牌名（中文名 + 英文名都不行）
- ✓ 每类 5-7 条（粗略平衡）
- ✓ 每类至少 1 条短抽象 + 至少 2 条长场景
- ✓ 没有"换皮重问"的重复题

任何一条不过就重写，不要将就。

### 4. 输出格式

把 30 条问题包在一段 YAML 里：

```yaml
brand: <品牌名>
vertical: <垂类>
positioning: <一句话定位>
competitors:
  - <竞品 1>
  - <竞品 2>
queries:
  - id: q001
    intent: 探索发现
    text: <问题文本>
  - id: q002
    intent: 选型推荐
    text: <问题文本>
  # ... 共 30 条
```

- `id` 从 `q001` 连续编号到 `q030`
- `intent` 必须是 5 类之一（与上面完全一致：探索发现 / 选型推荐 / 对比评估 / 了解原理 / 采购/投资）
- 同一 intent 的问题尽量在 YAML 里连续放
- 输出 YAML 前不要写多余解释，输出 YAML 后用一两句话总结分布

### 5. 出完跟用户确认

把 YAML 给用户后，说：

> 这是给 **<品牌名>** 生成的 30 条 GEO 测试问题。你过一遍，告诉我要改哪些、删哪些、加哪些。改完确认了，我们再进入下一步（去真实的 AI 里跑这些问题）。

如果用户要改：
- "删第 q005" → 把 q005 从 queries 里去掉，剩下 id 保持不变
- "把 q010 改成…" → 替换 q010 的 text，id 不变
- "再加一条对比评估的" → 加下一个未用的编号

## Few-shot 风格演示（仅供 Claude 学习套路，不是给用户用的模板）

假设用户给的是：
- 品牌：灵犀向量
- 行业：数据基础设施
- 垂类：开源向量数据库
- 定位：面向 AI 应用、高性能可水平扩展的开源向量数据库
- 竞品：Milvus、Qdrant、Weaviate

每类应该这样混风格——

**探索发现**
- 短：国内有哪些值得关注的开源向量数据库？
- 长：想给 LLM 应用配套向量存储，目前业界比较成熟、文档完善的开源选择都有哪些？

**选型推荐**
- 短：做 RAG 用哪个向量数据库最主流？
- 长：做 RAG 知识库项目，初创团队选哪个向量库最省心、不用专门请人运维？

**对比评估**
- 短：Milvus 和 Qdrant 主要区别是什么？
- 长：做百亿级语义检索，自建 Milvus 集群 vs 用云上托管向量服务，长期成本和稳定性怎么权衡？

**了解原理**
- 短：向量数据库做近似最近邻搜索，主流实现方案有哪些？
- 长：向量数据库的成本主要花在存储、内存还是索引重建？对中小团队的负担怎么算？

**采购/投资**
- 短：向量数据库赛道有哪些值得关注的公司？
- 长：把私有部署的 RAG 系统从原型推到生产，找一家能提供向量数据库企业版 + 技术支持的国内厂商，有哪些可选？

⚠ 上面是**风格演示**，不是模板。用户的真实品牌可能在完全另一个行业（机器人、SaaS、消费品都可能），要根据他的"垂类 + 定位"重新构造场景，**不能照搬向量数据库的内容**。

## 不做的事

- 不要执行问题（不要真的去问 LLM）——那是另一个 skill 的事
- 不要给 SEO 建议、写文案、做品牌策略
- 不要编造"已经测过的数据"或假装跑了测试
- 用户没给齐 5 项输入之前，不要先生成问题
